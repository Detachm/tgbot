# LBank 合约交易助手 Bot 架构规划

## 目标

本项目要做的不是普通喊单群机器人，而是一个面向 LBank 合约用户的 `Telegram Bot（电报机器人）`。它分两层服务用户：未绑定账户的用户获得信号、教程、注册链接；绑定账户的用户获得余额查询、持仓查询、仓位计算、确认式一键开仓和平仓。

当前第一目标是做出完整可演示的 demo（演示版本）：用户侧反馈、管理员侧反馈、信号流转、仓位计算、确认开仓、平仓和错误提示都要完整闭环；但 demo 阶段不向 LBank 发起真实下单和平仓请求。所有交易执行结果先由本地模拟执行器生成，并在用户消息、订单记录和审计日志中明确标注为模拟成交。

核心原则是：用户操作足够短，交易行为足够明确，系统错误必须显式暴露。项目不做兼容式猜测，不做默认回退，不做静默处理。

官方能力依据：

- Telegram Bot API 支持消息、按钮、回调和 `webhook（服务器回调）`：https://core.telegram.org/bots/api
- LBank 合约 API 支持申请 API Key，并提供账户、下单、撤单、订单、账户信息等私有接口：https://www.lbank.com/en-US/docs/contract.html

## 产品分层

未绑定账户用户看到的是轻量引流层。Bot 提供最新信号、历史信号、开通合约教程、注册链接和打开交易页按钮。这个层的目标是降低新用户理解成本，并记录来源和点击。

绑定账户用户看到的是交易助手层。Bot 读取用户合约账户余额、持仓、订单状态，根据管理员信号和用户风控配置自动计算保证金、杠杆、数量、止盈止损，并在用户明确确认后执行交易。demo 阶段的执行是模拟执行，不产生真实交易所订单。

交易助手层不提供默认自动跟单。后续可以预留自动跟单接口，但第一版只做确认式交易。

## 核心体验

用户第一次进入 Bot：

```text
欢迎使用 LBank 合约交易助手

[一键注册 LBank]
[绑定 LBank 账户]
[查看今日信号]
[开通合约教程]
```

未绑定用户收到信号：

```text
BTC/USDT 做多

入场：市价
止盈：63800
止损：62200
建议风险：账户 3%
建议杠杆：5x

[一键注册 LBank]
[打开 BTC 合约]
[绑定账户后自动算仓位]
```

绑定用户收到信号：

```text
BTC/USDT 做多

可用余额：1000 USDT
建议保证金：30 USDT
杠杆：5x
预计仓位：150 USDT
止盈：63800
止损：62200

[确认开多]
[调整仓位]
[取消]
```

用户点击确认后，Bot 先创建交易意图，再做最终风控校验，再调用交易执行器。demo 阶段使用模拟执行器，返回模拟成功或模拟失败；正式交易阶段再切换为 LBank 执行器。下单结果必须明确展示成功或失败原因，并在 demo 阶段明确显示“模拟成交”。

## 系统边界

系统只负责信号分发、账户读取、仓位计算、交易确认和下单执行。系统不承诺收益，不替用户做投资判断，不自动修改用户风险偏好。demo 阶段只做模拟交易执行，不触发真实交易所下单。

所有交易动作必须满足三个条件：信号存在、用户配置允许、用户本次明确确认。任何一个条件缺失，交易不得执行。

## 技术选型

服务端使用 `Node.js（服务端运行环境）` 和 `TypeScript（带类型的 JavaScript）`。Bot 框架使用 `grammY（电报机器人框架）` 或 `Telegraf（电报机器人框架）`，优先选择 grammY，因为它的中间件和会话模型更清晰。

数据库使用 `PostgreSQL（关系型数据库）`。交易、余额快照、用户配置、信号和审计日志都必须落库。缓存使用 `Redis（内存数据库）`，只用于限流、短期会话、幂等锁和任务队列状态，不作为事实来源。

后台任务使用 `BullMQ（队列任务库）`。推送信号、刷新订单、查询持仓、重试明确可重试的外部请求，都走队列。

部署使用单体服务优先，保留模块边界。第一版不拆微服务。拆早了会增加复杂度，不能带来真实收益。

## 模块划分

`bot` 模块负责 Telegram 消息、按钮、回调和命令处理。它只做输入输出编排，不直接写交易逻辑。

`user` 模块负责 Telegram 用户、绑定状态、语言、来源、权限和风险配置。

`exchange` 模块负责交易所接口抽象、LBank API 签名、请求、响应解析、错误映射。它是唯一允许访问 LBank 的模块。

`execution` 模块负责交易执行器选择。demo 阶段使用 `paper_execution（模拟执行器）`，正式交易阶段使用 `lbank_execution（LBank 真实执行器）`。两个执行器必须实现同一套接口，避免后续替换时改动 Bot 交互、风控和订单主流程。

`account` 模块负责余额、持仓、订单读取和账户快照。

`signal` 模块负责信号创建、校验、发布、撤回和历史查询。

`risk` 模块负责仓位计算、杠杆限制、单笔风险、日内风险、交易频率和禁止交易规则。

`trade` 模块负责交易意图、确认、下单、撤单、平仓、订单状态同步。

`audit` 模块负责审计日志。所有 API 绑定、配置变更、交易确认、下单请求、下单响应都必须记录。

`admin` 模块负责管理员发信号、查看用户统计、查看错误事件和人工停机。

## 数据模型

核心表如下：

```text
users
api_credentials
user_risk_profiles
signals
signal_deliveries
trade_intents
orders
positions
account_snapshots
referral_events
audit_logs
system_events
```

`api_credentials` 只保存加密后的 API Key 和 API Secret。密钥解密只能发生在调用 LBank 前的极小范围内，不能进入日志，不能进入错误对象，不能返回给前端或 Bot 消息。

`trade_intents` 是交易前的事实记录。用户点击确认前，系统只创建意图，不直接下单。确认后，系统用 `intent_id（意图编号）` 做幂等键，防止重复点击导致重复开仓。

`audit_logs` 是不可变日志。任何审计记录只追加，不更新，不删除。

`orders` 必须区分模拟订单和真实订单。建议字段包含 `execution_mode（执行模式）`，取值为 `paper（模拟）` 或 `live（真实）`。demo 阶段所有订单都必须写为 `paper`。

`signals` 必须保存结构化字段，而不是只保存文本。第一版至少包含交易对、方向、入场类型、入场价、市价或限价、止盈、止损、建议风险、建议杠杆、状态、创建人和发布时间。信号状态至少包含 `draft（草稿）`、`published（已发布）`、`cancelled（已撤回）`、`expired（已过期）`。

## 交易流程

管理员创建信号时，系统先校验字段完整性。交易对、方向、入场类型、止盈、止损、建议风险、建议杠杆缺一不可。校验失败直接拒绝创建。

信号发布后，未绑定用户收到引流版消息，绑定用户收到计算版消息。绑定用户的计算依赖最新账户快照；如果账户快照过期，系统先刷新账户。刷新失败则不展示确认开仓按钮，只展示明确错误。

用户点击确认开仓后，系统按顺序执行：

```text
读取交易意图
校验意图未过期
刷新账户余额和持仓
执行风控校验
生成标准下单请求
写入待提交订单
调用交易执行器
写入执行器响应
推送结果给用户
进入订单状态同步
```

任一步失败都必须终止流程，并把失败原因写入 `system_events` 和用户可读消息。

demo 阶段的交易执行器必须是模拟执行器。模拟执行器不读取 API Secret，不调用 LBank 下单接口，只根据标准下单请求生成本地模拟订单号、模拟成交价、模拟状态和模拟失败原因。用户消息必须明确出现“模拟成交”或“模拟失败”，不能让用户误以为已经真实下单。

正式交易阶段切换到 LBank 执行器前，必须补齐真实交易状态机、交易所订单号或客户端订单号、未知状态恢复、限速和人工介入流程。

## 风控规则

第一版必须内置这些规则：

单笔保证金不能超过用户配置比例。默认建议 3%，用户可以降低或提高，但系统有平台上限。

杠杆不能超过平台上限。第一版建议平台上限设为 10x，即使信号写更高也拒绝执行。

同一交易对同方向不能重复开仓，除非用户显式开启允许加仓。

每日交易次数必须有限制。默认 10 次。

每日最大计划风险必须有限制。默认账户权益 15%。

API Key 检测到提现权限时拒绝绑定。若 LBank 无法通过接口判断提现权限，则绑定流程必须要求用户截图或明确确认，并在文案中强提示“必须关闭提现权限”。不能假设安全。

## 错误策略

系统错误分三类：

用户错误：例如未绑定账户、余额不足、参数超出限制。直接给用户明确提示。

交易所错误：例如签名失败、权限不足、交易对不存在、下单被拒绝。保留原始错误码，映射成中文说明。

系统错误：例如数据库失败、队列失败、未知异常。用户只看到“系统错误，订单未提交”或“状态未知，请联系客服”，管理员收到完整事件。

不允许静默失败。不允许把未知错误当成成功。不允许在下单失败后自动改参数重试。涉及交易的重试只能用于查询状态，不能用于重复提交订单。demo 阶段的模拟失败也必须走同样的错误展示和审计路径，不能因为是模拟功能就静默处理。

## 幂等与一致性

所有交易提交必须使用业务幂等键。格式建议：

```text
tg:{telegram_user_id}:intent:{intent_id}
```

用户重复点击确认，只能返回同一个交易结果，不能创建第二笔订单。

下单前写入本地订单记录，状态为 `pending_submit（等待提交）`。demo 阶段调用模拟执行器成功后更新为 `paper_filled（模拟成交）` 或 `paper_rejected（模拟拒绝）`。正式交易阶段调用 LBank 成功后更新为 `submitted（已提交）`。如果网络中断导致状态未知，订单进入 `unknown（未知）`，只能通过查询交易所订单状态恢复，不能重新下单。

账户余额和持仓以 LBank 为准。本地快照只用于展示和计算前置参考，下单前必须重新读取。

## 安全设计

API Secret 使用应用级加密。生产环境密钥放在云服务密钥管理或独立环境变量中，不写入代码和数据库。

Bot 管理员权限使用白名单 Telegram 用户编号。管理员命令必须记录审计日志。

所有回调数据只放短编号，不放完整交易参数。完整参数从数据库读取，避免用户篡改按钮数据。

敏感操作需要短时确认窗口。交易意图默认 60 秒过期。过期后用户必须重新生成订单。

日志默认脱敏。禁止记录 API Secret、签名、完整请求头。

## 性能设计

第一版的性能瓶颈不会在计算，而在 Telegram 推送和 LBank API 限速。设计上采用队列削峰。

信号发布写入数据库后立即返回管理员确认，推送任务异步执行。推送失败要记录每个用户的失败原因，不能影响其他用户。

账户刷新按用户触发，不做全量轮询。只有用户打开余额、收到可交易信号、点击确认时才刷新。

订单状态同步使用短周期任务，只跟踪未完成订单。已完成订单不再频繁查询。

## 接口预留

交易执行器接口必须从第一版开始保留。Bot、风控和订单流程只能依赖统一接口，不能直接依赖 LBank 下单接口。建议形态：

```text
submitOrder(request) -> order_result
closePosition(request) -> close_result
getOrderStatus(order_id) -> order_status
```

demo 阶段的实现是 `paper_execution`。真实交易阶段新增 `lbank_execution`，并通过配置开关选择执行模式。

后续自动跟单通过 `strategy` 模块接入。它只能创建交易意图，不能绕过 `risk` 和 `trade` 模块直接下单。

后续 Mini App 通过同一套后端接口接入。Mini App 只能作为更好的界面，不拥有独立交易逻辑。

后续多交易所通过 `exchange` 抽象接入。接口形态必须统一为账户、持仓、订单、下单、撤单、平仓、行情精度。不同交易所的差异只能在适配器内部处理。

后续 VIP、返佣、分销通过 `billing` 和 `referral` 模块接入，不污染交易流程。

## mock 与实装标记

开发过程中，每个功能必须标明当前状态，避免把模拟能力误认为真实能力。标记规则如下：

```text
real：已经接入真实外部能力或真实持久化，行为可用于生产路径
mock：本地模拟或固定返回，用于演示完整体验，不能用于生产路径
partial：部分真实、部分模拟，必须说明边界
planned：尚未实现
```

第一版 demo 的建议标记：

```text
Telegram Bot 基础交互：real
用户绑定状态：real
管理员发信号：real
信号结构化保存：real
未绑定用户信号推送：real
点击和来源统计：real
审计日志：real
仓位计算：real
基础风控：real
管理员停机开关：real
LBank API Key 权限检测：partial
余额查询：partial
持仓查询：partial
确认式开仓：mock
一键平仓：mock
订单状态同步：mock
真实 LBank 下单：planned
真实 LBank 平仓：planned
未知状态恢复：planned
```

所有用户可见的 mock 交易结果都必须带“模拟”字样。所有后台记录也必须写入 `execution_mode=paper`，方便后续清理、统计和切换真实交易。

## 验收标准

用户未绑定账户时，可以完整接收信号并跳转注册。

用户绑定账户后，可以查询余额和持仓。

管理员发布一条信号后，未绑定用户和绑定用户收到不同版本内容。

绑定用户点击确认开仓后，只产生一笔交易意图和最多一笔本地模拟订单。demo 阶段不得产生真实交易所订单。

余额不足、API 权限错误、交易对错误、风控拒绝，都必须明确展示。

系统不能在任何未知状态下重复提交订单。demo 阶段虽然不真实下单，也必须验证重复点击只返回同一笔模拟订单结果。

管理员可以一键关闭新开仓，关闭后用户只能查账户和看信号。

所有关键行为都能在审计日志中还原。

## 最优实施顺序

先实现 Telegram Bot、数据库和用户状态。再实现信号创建、信号推送、点击统计和审计日志。然后实现仓位计算、基础风控、模拟开仓和模拟平仓，让 demo 先形成完整闭环。LBank API 签名和只读账户查询可以并行推进，但不阻塞模拟交易 demo。最后再接真实确认式开仓和平仓。

顺序不能反过来。没有账户读取、风控校验、真实交易状态机和审计日志，就不应该开放真实交易。demo 阶段可以先用模拟执行器验证完整体验，但必须在界面和后台记录里明确标注模拟。
